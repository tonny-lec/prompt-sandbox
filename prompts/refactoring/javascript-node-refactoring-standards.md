<role>
You are a senior refactoring guide, Node.js/JavaScript architect, technical writer, and repository standards designer.
Your job is to create a practical refactoring documentation set that Codex and human developers can use as an ongoing implementation and review reference.
</role>

<goal>
Create an operational documentation set for JavaScript / Node.js refactoring.
This documentation must not be a generic summary.
It must function as:
1. a decision standard for identifying unnecessary complexity and redundancy,
2. a practical implementation guide for refactoring code safely,
3. a review guide for pull requests,
4. an operational handbook that can be maintained over time.

The documentation will be used by Codex itself as a long-term reference during future implementations and refactorings.
</goal>

<context>
We have observed that code generated by Codex can become significantly more verbose than the intended design.
We want to establish a clear internal refactoring standard so that:
- Codex can refactor existing code better,
- Codex can use the same standard in future implementations,
- humans can review changes using the same criteria.

Target stack:
- JavaScript
- Node.js

Focus:
- reduce verbosity without harming readability,
- improve separation of concerns,
- reduce duplication,
- align async/error-handling patterns,
- make code easier to test and maintain,
- document exceptions and tradeoffs,
- define operational usage, not just principles.
</context>

<source-policy>
Use primary and authoritative sources first.
Priority order:
1. Node.js official documentation
2. ESLint official documentation
3. MDN documentation
4. Martin Fowler refactoring catalog
5. well-known JavaScript style guides and practical best-practice references

For every major statement:
- mark whether it is Verified or Inference
- include source links or citations when verified
- if something cannot be firmly verified, explicitly label it as Inference
</source-policy>

<requirements>
Each file must be practical and reusable.
Do not write high-level theory only.

For each document, include:
- purpose of the document
- when to use it
- concrete decision criteria
- step-by-step usage guidance
- examples where applicable
- anti-patterns or failure cases where applicable

Detailed requirements by file:

1. README.md
- explain the overall purpose of the refactoring standards
- explain which file to consult at each stage:
  - before implementation
  - during refactoring
  - before PR
  - during review
  - when updating the standards
- provide a recommended reading order

2. 01-principles.md
- define refactoring goals
- define what counts as “too verbose” in JS/Node.js
- define principles such as single responsibility, duplication reduction, clarity over cleverness, behavioral preservation, testability, and explicit error handling
- separate Verified vs Inference content clearly

3. 02-js-node-guidelines.md
- define JS/Node.js-specific refactoring guidance
- include async/await usage, promise handling, callback reduction, module boundaries, side-effect isolation, validation placement, logging responsibility, configuration handling, and dependency boundaries
- include examples of bad vs better patterns

4. 03-code-smells-and-actions.md
- create a table of code smells with:
  - smell name
  - symptoms
  - why it is harmful
  - recommended refactoring actions
  - priority level
  - cautions / tradeoffs
- focus on code smells common in JavaScript / Node.js projects

5. 04-refactoring-checklist.md
- create operational checklists for:
  - before refactoring
  - during refactoring
  - before commit
  - before pull request
- checklists must be concise and actionable

6. 05-safety-and-tests.md
- explain how to refactor safely
- define what tests or verification steps should exist before changing behavior-sensitive code
- explain test priorities for pure logic, integration points, async code, filesystem/network boundaries, and error cases
- include minimum verification guidance

7. 06-review-playbook.md
- define a review order and review questions
- reviewers should be able to decide:
  - whether behavior changed unintentionally
  - whether complexity decreased
  - whether responsibilities became clearer
  - whether naming improved
  - whether future modifications became easier
- include review comment templates

8. 07-metrics.md
- define measurable indicators for refactoring quality
- include examples such as function length, duplication count, lint issues, cyclomatic complexity, review issue recurrence, test coverage delta, and hotspot tracking
- explain which metrics are useful and which are misleading

9. 08-examples-before-after.md
- include multiple concrete before/after examples in JavaScript / Node.js
- examples should cover:
  - long function decomposition
  - callback/promise cleanup
  - duplicated validation removal
  - mixed I/O and business logic separation
  - improving error handling
- for each example, explain why the after-version is better

10. 09-exceptions-and-tradeoffs.md
- document when the general rules should not be applied blindly
- include performance-sensitive paths, low-level Node.js cases, temporary migration code, and team consistency considerations
- explain how to record exceptions

11. 10-maintenance-policy.md
- define how this standards set should be updated
- define when to add new examples
- define how to retire outdated guidance
- define how to record lessons learned from real refactoring work
- include a lightweight operating model for continuous maintenance
</requirements>
