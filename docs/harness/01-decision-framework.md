# Decision Framework

## Purpose

どの課題にどの技術を当てるかを決めるための判断基準です。

## When To Use

- 新しいハーネスを作る前
- 既存の実験を実 API へ移行する前
- SDK から framework へ移すべきか迷ったとき

## Decision Criteria

| 観点 | SDK / batteries-included が向く条件 | Framework / 自前構築が向く条件 |
| --- | --- | --- |
| 利用者数 | 個人利用、少人数、研究用途 | 複数ユーザー、本番運用 |
| 速度 | 数秒の待ちが許容される | 低遅延が重要 |
| コスト | 多少の token 増加を許容 | token 管理を厳密にしたい |
| 制御性 | 組み込み能力を優先 | 履歴、状態、分岐を細かく制御したい |
| 観測性 | 簡易ログで足りる | 詳細なメトリクスや再現性が必要 |
| 立ち上がり | すぐ試したい | 初期コストを払ってでも長期運用したい |

## Step-By-Step Guidance

1. そのハーネスを誰が使うかを決める
2. 速度とコストの許容値を決める
3. 会話履歴や中間状態を自前管理したいか確認する
4. `handoff`, `skills`, `MCP`, `RAG` が MVP に必要かを切り分ける
5. 最も軽い選択肢から始め、必要時のみ下位レイヤーへ降りる

## Default Policy In This Repo

- 研究開始時は軽量な mock / SDK 寄りの流れで始める
- 次のどれかを満たしたら framework 側の設計を追加する
  - 応答時間の比較が必要
  - 追跡すべき状態が増えた
  - 履歴や評価の再現性を高めたい
  - 複数の provider / runtime を横並び比較したい

## Failure Cases

- 本番要件なのに SDK の便利さだけで選ぶ
- 研究段階なのに最初から複雑な framework を組み始める
- 速度の課題を prompt 修正だけで解決しようとする
